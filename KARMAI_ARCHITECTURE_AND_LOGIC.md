# KarmAI - System Architecture and Logic Guide

This document provides a comprehensive overview of the KarmAI project, detailing the tech stack, data schemas, data collection, and internal calculation logic. It is structured to provide an LLM or developer with complete context for writing scripts, implementing features, or extending the application.

## 1. Project Overview

KarmAI is a gamified, mobile-first web application that bridges real-world actions with digital rewards. It incentivizes positive behavior (Karma), connects college students, and provides AI-powered localized discovery.

## 2. Core Tech Stack

- **Frontend:** React, TypeScript, Vite, Tailwind CSS (Custom Glassmorphism design system).
- **Database & Auth:** Supabase (PostgreSQL, `pgvector`, Supabase Auth).
- **Backend:** Python, FastAPI, optimized for AMD GPU execution (handling semantic embeddings and local LLM inference).

---

## 3. Data Models: What Information is Collected?

### A. User Profiles (`profiles` table)

This table acts as the unified identity matrix for gamification and authentication.

- **`id` (UUID):** Maps exactly to Supabase Auth's user ID.
- **`full_name` (Text):** The user's displayed name.
- **`avatar_url` (Text):** Profile image pointer.
- **`college` / `institution` (Text):** Localized context (e.g., "K.K. Wagh Institute", "VJTI").
- **`karma_score` (Integer):** Total accumulated gamification points.
- **`level` (Integer):** Tier classification derived from karma.

### B. Map & Venues (`locations` table)

Powers the `ExploreScreen` and RAG AI engine.

- **`id` (UUID):** Unique location identifier.
- **`name` & `category` (Text):** Descriptive tags (e.g., "Library", "Cafe").
- **`lat` / `lng` (Numeric):** Geospatial coordinates for rendering `MapCore`.
- **`description` (Text):** Human-readable context about the venue.
- **`embedding` (Vector 384):** Mathematical representation of the description, generated by the backend, used for semantic AI matching.
- **`popularity` (Integer):** A score that determines baseline sorting in the UI.

### C. Activities & Check-ins (`check_ins`) _(Conceptual/In-Progress)_

- **`user_id` & `location_id`:** Relational linking.
- **`proof_type`:** GPS validation, uploaded image, or QR scan.
- **`ai_verification_status`:** Whether the AMD backend approved the visual/text proof.

## 4. Core Logic & Calculations

### A. Gamification (Karma & Levels)

How does a user earn points?

1.  **Check-ins:** Registering presence at a location (Standard: +10 Karma).
2.  **Social/Community Actions:** Verified actions like cleaning a campus area or helping a club (Premium: +50 to +100 Karma).

**Calculation Engine:**
When an action is logged, the script must query the existing profile, increment the `karma_score`, and check if the user crosses the next level threshold.

- _Example Formula:_ Next level required Karma = `(Current Level * 100) * 1.5`

### B. RAG (Retrieval-Augmented Generation) & Discovery

How are locations recommended?

1.  **Input:** User asks the "AI Concierge" something (e.g., "Where is a quiet place to study near VJTI?").
2.  **Vectorization:** The backend script passes this text to a local embedding model (generating a 384-dimensional float array).
3.  **Calculation (Vector Search):** Supabase `pgvector` calculates the Cosine Distance (`<=>`) between the user's query vector and all `locations.embedding`.
4.  **Retrieval:** The closest top 5 vectors are returned, giving the app highly localized, semantic matches instead of basic keyword searches.

### C. Authentication & App State Flow

1.  **Global Identity:** The frontend uses `supabase.auth.onAuthStateChange` injected into `src/app/App.tsx`.
2.  **State Sync:** Whenever auth state changes, `fetchUserProfile()` runs, grabbing the user's row from `public.profiles`.
3.  **Prop Drilling:** This `UserProfile` JSON is passed globally so that the `LeaderboardScreen` can highlight the active user, and the `ExploreScreen` can customize defaults.

## 5. Integration Notes for AI Scripting

When writing specific integration scripts (like trigger functions, batch updates, or automated testing):

- **Data Updates:** Always use the authenticated Supabase client (`@supabase/supabase-js`) passing through proper RLS (Row Level Security) or use a Service Role Key for backend cron jobs.
- **Testing Coverage:** The database currently features Indian/Mumbai contextual data (e.g., Marine Drive, IIT Bombay, K.K. Wagh) specifically seeded with real math vectors for prompt testing.
- **Component Sync:** Any backend script modifying a user's `karma_score` directly in Supabase will automatically reflect in the frontend upon the next component remount or active realtime subscription.
